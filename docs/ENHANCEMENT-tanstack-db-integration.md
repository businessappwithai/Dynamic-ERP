# Enhancement: TanStack DB + ElectricSQL Integration for Generated Applications

## Document Information
- **Title**: TanStack DB + ElectricSQL Integration — Real-Time Offline-First Data Layer for AI-Generated Applications
- **Date**: 2026-05-16
- **Status**: Design Specification
- **Version**: 2.0

## Executive Summary

This enhancement specifies integrating **TanStack DB** with **ElectricSQL** as a reactive, offline-first data layer for applications generated by the app-with-ai-tanstack platform. This combination enables:

1. **Compiere Dictionary System (sys_*)** — Real-time synchronization of application metadata (sys_table, sys_column, sys_field) across all clients with offline support
2. **Collaborative Entity Management** — Multiple users can modify entities, forms, and field configurations simultaneously with conflict-free sync
3. **Offline-First Business Logic** — Users can continue working offline while queued mutations sync when reconnected
4. **Real-Time Dashboard and Admin UI** — System configuration changes propagate instantly across all connected users
5. **Type-Safe Reactive Queries** — TanStack DB's live queries ensure UI reactivity without manual subscription management

**Key Innovation**: ElectricSQL bridges the gap between PostgreSQL (backend authority) and local SQLite (client cache), enabling true real-time sync without WebSocket polling or eventual consistency issues.

## Problem Statement: Dictionary-Driven Applications at Scale

Generated TanStack Start/NestJS applications follow the **Compiere dictionary pattern**, where application metadata (sys_table, sys_column, sys_field) drives:
- Dynamic CRUD forms and tables
- Field-level validation rules
- UI layout and visibility constraints
- Business rule evaluation

**Current limitations:**

| Problem | Impact |
|---------|--------|
| Dictionary changes require full page reload | Users experience interruptions when admins modify schemas |
| Multiple admins editing sys_* tables conflict silently | Last-write-wins causes lost changes; manual conflict resolution needed |
| No offline access to system configuration | Power users can't work if connectivity drops |
| CRUD operations on dictionary tables trigger N+1 queries | Fetching entities with field metadata becomes slow as sys_field table grows |
| System configuration changes don't propagate instantly | New fields/validations require app restart or hard refresh |

## CRITICAL: Role-Based Access Control for sys_* Definitions

**⚠️ SECURITY REQUIREMENT**: ElectricSQL shapes for sys_* tables MUST be filtered by user role. Each user only syncs dictionary definitions they are authorized to view/modify. **Unauthorized users must NEVER see other users' or roles' custom field definitions, validation rules, or entity schemas.**

### Why This Matters

In a multi-user system, sys_* definitions are sensitive configuration:
- **sys_field visibility rules** — hiding certain fields from junior users reveals organizational structure
- **Custom validation rules** — reveal business logic and constraints
- **Field permissions** — expose which roles can edit what
- **Calculated field formulas** — proprietary business logic
- **Lookup relationships** — may expose confidential master data sources

**Scenario**: User A (sales) should NOT see User B (finance)'s custom GL account fields or validation rules.

### Implementation: ElectricSQL Shapes with RBAC Filters

```typescript
// packages/generator/templates/tanstack-start-nestjs/backend/src/lib/electric-sql/shapes.ts

import { Database } from 'kysely'
import { getCurrentUser } from '@/lib/better-auth'

/**
 * CRITICAL: Shape subscriptions MUST include WHERE filters for user authorization
 * Failure to filter = data leakage of other users' custom definitions
 */
export async function subscribeToUserDefinitions(
  electric: ElectricClient,
  userId: string,
  userRoles: string[]
) {
  // ✅ CORRECT: Only subscribe to definitions user can access
  
  // User's own custom fields
  await electric.db.raw(
    `SELECT * FROM sys_field 
     WHERE created_by = ? OR role_accessible IN (${userRoles.map(() => '?').join(',')})`,
    [userId, ...userRoles]
  )
  
  // User's accessible tables
  await electric.db.raw(
    `SELECT * FROM sys_table 
     WHERE created_by = ? OR role_accessible IN (${userRoles.map(() => '?').join(',')})`,
    [userId, ...userRoles]
  )
  
  // User's accessible columns
  await electric.db.raw(
    `SELECT * FROM sys_column 
     WHERE table_id IN (
       SELECT id FROM sys_table 
       WHERE created_by = ? OR role_accessible IN (${userRoles.map(() => '?').join(',')})
     )`,
    [userId, ...userRoles]
  )
  
  // User's accessible rules
  await electric.db.raw(
    `SELECT * FROM sys_rule 
     WHERE created_by = ? OR role_accessible IN (${userRoles.map(() => '?').join(',')})`,
    [userId, ...userRoles]
  )
}

/**
 * ❌ INCORRECT: This leaks all sys_field definitions to all users
 */
export async function subscribeToUserDefinitions_UNSAFE(
  electric: ElectricClient
) {
  // DON'T DO THIS!
  await electric.db.raw(`SELECT * FROM sys_field`) // LEAKS ALL FIELDS!
  await electric.db.raw(`SELECT * FROM sys_rule`)  // LEAKS ALL RULES!
  // User A can now see User B's custom fields and business rules
}
```

```typescript
import { electric } from '@/lib/electric-sql/client'

export async function initializeUserShapes() {
  const { user, roles } = useAuth()
  
  if (!user) return // Not authenticated
  
  // Call backend to get authorized shapes
  const authorizedShapes = await fetch('/api/electric-sql/shapes', {

    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      userId: user.id,
      roles: roles,
    }),
  }).then(r => r.json())
  
  // Subscribe only to authorized sys_* definitions
  for (const shape of authorizedShapes) {
    await electric.db.raw(shape.query, shape.params)

  // Collections now only contain authorized definitions
  // User A cannot see User B's custom fields
}
```

### Backend API Endpoint for Shape Authorization

```typescript
// packages/generator/templates/tanstack-start-nestjs/backend/src/routes/api/electric-sql/shapes.ts

import { createAPIFileRoute } from '@tanstack/start/api'
import { getCurrentUser } from '@/lib/better-auth'
import { db } from '@/lib/database'

export const Route = createAPIFileRoute('/api/electric-sql/shapes')({
  POST: async ({ request }) => {
    const user = await getCurrentUser(request)
    if (!user) return new Response('Unauthorized', { status: 401 })
    
    const { userId, roles } = await request.json()
    
    // Only return shapes for authenticated user (no leakage)
    if (userId !== user.id) {
      return new Response('Unauthorized', { status: 403 })
    }
    
    // Query database for user's authorized definitions
    const authorizedTables = await db
      .selectFrom('sys_table')
      .selectAll()
      .where((eb) =>
        eb.or([
          eb('created_by', '=', user.id),
          eb('role_accessible', '@>', JSON.stringify(roles)),
          eb('is_public', '=', true),
        ])
      )
      .execute()
    
    const authorizedFields = await db
      .selectFrom('sys_field')
      .selectAll()
      .where((eb) =>
        eb.or([
          eb('created_by', '=', user.id),
          eb('role_accessible', '@>', JSON.stringify(roles)),
          eb('is_public', '=', true),
        ])
      )
      .execute()
    
    // Return ElectricSQL-compatible shape queries
    return new Response(
      JSON.stringify({
        authorizedShapes: [
          {
            query: `SELECT * FROM sys_table WHERE id IN (${authorizedTables.map(t => `'${t.id}'`).join(',')})`,
            params: [],
          },
          {
            query: `SELECT * FROM sys_field WHERE id IN (${authorizedFields.map(f => `'${f.id}'`).join(',')})`,
            params: [],
          },
        ],
      }),
      { headers: { 'Content-Type': 'application/json' } }
    )
  },
})
```

## Proposed Architecture: TanStack DB + ElectricSQL

```
Browser UI (React) — TanStack Start
   ↓ useLiveQuery()
TanStack DB Collections (in-memory reactive tables)
   ↓ ElectricSQL sync (real-time bidirectional)
ElectricSQL Client (local SQLite buffer)
   ↓ WebSocket + shape subscriptions
ElectricSQL Server (PostgreSQL logical replication)
   ↓ Logical Decoding
PostgreSQL (authority)
   • sys_table, sys_column, sys_field
   • bus_entity_* tables
   • auth_*, user tables
```

### Architecture Tiers

**Tier 1: ElectricSQL (Real-Time Sync)**
- Client-side local SQLite maintained by ElectricSQL
- Automatic bidirectional sync via logical replication
- Offline-first: queued mutations persist locally until reconnect
- Conflict-free: Last-write-wins, but tracks causality

**Tier 2: TanStack DB (Reactive Collections)**
- Wraps ElectricSQL local SQLite as reactive collections
- Provides `useLiveQuery()` hooks for React components
- Enables optimistic mutations with rollback on error
- Type-safe through Zod schemas

**Tier 3: TanStack Start Server (Authority)**
- PostgreSQL as single source of truth
- NestJS/Fastify business logic, validation, authorization
- Serves initial data; ElectricSQL maintains cache thereafter

### Key Concepts

| Concept | Description |
|---------|-------------|
| **Shape** (ElectricSQL) | Subscription to a server-side view (e.g. `SELECT * FROM sys_table WHERE tenant_id = ?`) |
| **Collection** (TanStack DB) | In-memory reactive table backed by local SQLite |
| **useLiveQuery** | React hook subscribing to live collection query results |
| **Optimistic mutation** | Write to local DB immediately; confirm/rollback on server validation |
| **Sync conflict** | Rare; ElectricSQL handles via causality tracking; admin UI for manual resolution |

### 2. Dynamic Business Entity CRUD Forms

**Current state** (generated TanStack Start frontend):
- Form renders sys_field metadata fetched from server
- Field visibility / validation rules require separate API calls
- Form submission must re-validate against potentially stale field schema
- Multi-user scenarios: user A modifies field; user B's form still shows old validation rules

**With TanStack DB + ElectricSQL:**

```typescript
// Generated form component template
import { useLiveQuery } from '@tanstack/db'
import { sysFieldCollection, busEntityDataCollection } from '@/lib/tanstack-db/collections'

export function DynamicEntityForm({ entityType, recordId }: Props) {
  // Real-time schema from synced sys_field
  const fields = useLiveQuery(() =>
    sysFieldCollection
      .filter(f => f.table === entityType && f.isDisplayed)
      .sortBy('seqNo')
  )
  
  // Real-time entity data
  const [data, setData] = useLiveQuery(() =>
    busEntityDataCollection
      .filter(d => d.entityType === entityType && d.id === recordId)
  )
  
  async function updateField(fieldName: string, value: unknown) {
    // Optimistic write to local collection
    const fieldDef = fields.find(f => f.columnName === fieldName)
    
    // Client-side validation using live schema
    if (fieldDef?.isRequired && !value) {
      showError(`${fieldDef.displayName} is required`)
      return
    }
    
    // Write locally first
    await busEntityDataCollection.update(recordId, {
      [fieldName]: value,
      updatedAt: new Date(),
    })
    
    // ElectricSQL syncs in background; server validates and broadcasts conflict if any
  }
}

// Register ElectricSQL shape subscriptions (one-time setup)
export async function initializeEntityShapes(tenantId: string, entityTypes: string[]) {
  for (const entityType of entityTypes) {
    // Subscribe to sys_field changes for this entity
    await electric.db.raw(
      `SELECT * FROM sys_field WHERE table_name = ? AND tenant_id = ?`,
      [entityType, tenantId]
    )
    
    // Subscribe to live entity data
    await electric.db.raw(
      `SELECT * FROM bus_${entityType} WHERE tenant_id = ?`,
      [tenantId]
    )
  }
}
```

**Benefits:**
- **Instant validation**: Form validation rules update in real-time as admins modify sys_field
- **Offline-first forms**: Users can continue editing even if network drops; forms sync on reconnect

### 3. Multi-Tenant Sys_ Queries with Scoped Subscriptions

**Current state**:
- Every sys_table fetch requires re-querying the backend
- Tenant isolation happens at API layer; no automatic filtering
- Switching tenants requires full data refresh

**With TanStack DB + ElectricSQL:**

```typescript
// Global sync configuration (generated into TanStack Start app)
import { useTenant } from '@/hooks/useTenant'
import { createElectricClient } from 'electric-sql/client'

export async function setupElectricSync(tenantId: string, userId: string, userRoles: string[]) {
  const electric = await createElectricClient({
    url: import.meta.env.VITE_ELECTRIC_URL || 'http://localhost:5133',
    auth: {
      token: await getAuthToken(), // From BetterAuth session
    },
  })
  
  // 🔒 SECURITY: Subscribe only to sys_* definitions user is authorized to view
  // Filters by: created_by, role_accessible, is_public
  const roles = userRoles.map(r => `'${r}'`).join(',')
  
  await electric.db.raw(
    `SELECT * FROM sys_table 
     WHERE tenant_id = ? 
     AND (created_by = ? OR role_accessible @> ARRAY[${roles}] OR is_public = TRUE)`,
    [tenantId, userId]
  )
  await electric.db.raw(
    `SELECT * FROM sys_column 
     WHERE tenant_id = ? 
     AND table_id IN (
       SELECT id FROM sys_table 
       WHERE (created_by = ? OR role_accessible @> ARRAY[${roles}] OR is_public = TRUE)
     )`,
    [tenantId, userId]
  )
  await electric.db.raw(
    `SELECT * FROM sys_field 
     WHERE tenant_id = ? 
     AND (created_by = ? OR role_accessible @> ARRAY[${roles}] OR is_public = TRUE)`,
    [tenantId, userId]
  )
  
  // Subscribe to user's business entities
  const tables = await sysTableCollection.all()
  for (const table of tables) {
    await electric.db.raw(
      `SELECT * FROM bus_${table.tableName} WHERE tenant_id = ?`,
      [tenantId]
    )
  }
  
  return electric
}

// Use in any component
function TenantSwitcher() {
  const { tenant, switchTenant } = useTenant()
  
  async function onTenantChange(tenantId: string) {
    // Unsubscribe old tenant shapes
    await stopElectricSync()
    
    // Re-subscribe to new tenant
    await setupElectricSync(tenantId, userId)
    
    // Collections automatically repopulate from ElectricSQL
    switchTenant(tenantId)
  }
}
```

**Benefits:**
- **Automatic tenant isolation**: ElectricSQL shapes enforce tenant_id filtering at the logical replication level
- **Instant tenant switching**: No full page reload; just unsubscribe old shapes, subscribe new ones
- **Multi-tenant safety**: Database-enforced filters prevent accidental cross-tenant data leaks

## Sync Strategy: ElectricSQL Shapes + Server Validation

ElectricSQL provides automatic, conflict-free bidirectional sync. The recommended approach:

### ElectricSQL Sync Flow

```
User action (update sys_field)
   ↓
Write to local SQLite (via TanStack DB)
   ↓
ElectricSQL detects change
   ↓
Send to ElectricSQL server (WebSocket)
   ↓
ElectricSQL server applies to PostgreSQL
   ↓
Backend NestJS validates mutation
   ↓
If valid: mark in-sync; broadcast to other clients via logical replication
If invalid: rollback; send conflict message back to client
```

### Server-Side Validation Hooks (NestJS Backend)

```typescript
// packages/generator/templates/tanstack-start-nestjs/backend/src/modules/sys/sys.service.ts
import { Injectable } from '@nestjs/common'
import { Database } from 'kysely'

@Injectable()
export class SysService {
  constructor(private db: Database) {}

  async updateSysField(id: string, data: Partial<SysField>) {
    // Validate before accepting mutation
    if (data.isRequired && !data.displayName) {
      throw new Error('Required field must have display name')
    }

    // Update in PostgreSQL
    const updated = await this.db
      .updateTable('sys_field')
      .set(data)
      .where('id', '=', id)
      .executeTakeFirst()

    // ElectricSQL automatically replicates via logical decoding
    // All subscribed clients receive update instantly
    return updated
  }
}
```

### Conflict Resolution UI

```typescript
// Component that notifies user of conflicts (rare, but handled gracefully)
function SyncStatusBar() {
  const [conflicts, setConflicts] = useState<ConflictNotification[]>([])
  
  useEffect(() => {
    // Listen for conflicts from ElectricSQL
    electric.on('conflict', (conflict: ConflictEvent) => {
      setConflicts(prev => [...prev, {
        table: conflict.table,
        recordId: conflict.recordId,
        message: `Another user modified ${conflict.table}. Showing merged version.`,
        resolvedAt: new Date(),
      }])
    })
  }, [])
  
  return (
    <div className="sync-status">
      {conflicts.length > 0 && (
        <Alert variant="info">
          {conflicts.map(c => (
            <p key={c.recordId}>{c.message}</p>
          ))}
        </Alert>
      )}
    </div>
  )
}
```

**Key benefits of ElectricSQL over custom sync:**
1. **Logical replication** — Uses PostgreSQL's built-in CDC, not webhooks or polling
2. **Conflict-free merge** — Last-write-wins with causality tracking; no manual conflict resolution in most cases
3. **Offline queuing** — Mutations queue locally and replay on reconnect
4. **Type safety** — Schema inference from PostgreSQL; no separate sync contracts

## TanStack DB + ElectricSQL vs TanStack Query — Decision Matrix

In generated applications, the choice is clear by data type:

| Data Type | Storage | Use | Why |
|-----------|---------|-----|-----|
| **sys_table, sys_column, sys_field** | ElectricSQL + TanStack DB | Read-only via `useLiveQuery()` | Real-time updates, offline access, no pagination |
| **Business entities (bus_*)** | ElectricSQL + TanStack DB | CRUD forms, bulk edits | Multi-user sync, offline offline, validation rules |
| **Audit logs** | ElectricSQL + TanStack DB | Read-only via `useLiveQuery()` | Real-time compliance, offline access |
| **Rules engine cache** | ElectricSQL + TanStack DB | Read via `useLiveQuery()` | Live rule updates across clients |
| **One-off data fetch** | TanStack Query | e.g., get user profile by ID | Not suitable for ongoing sync |
| **Large paginated tables** | TanStack Query + Server-side pagination | Data tables with 10k+ rows | Avoid syncing entire table; keep pagination on server |
| **External API results** | TanStack Query | OpenAI completions, exports | Not database-backed; no sync needed |

**Architecture rule for generated apps:**
- **Small working sets (<5k rows)** → ElectricSQL shapes + TanStack DB collections
- **Large queryable tables** → TanStack Query with server-side pagination (cursor-based preferred)
- **Hybrid**: Use ElectricSQL for metadata (sys_*); TanStack Query for large entity lists

**Pagination strategy:**
- sys_* tables: No pagination (typically <100 rows)
- Audit logs: Cursor-based pagination with `limit 100` per fetch
- Business entities: Server-side pagination in grid views; ElectricSQL for single-record forms

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| **🔒 DATA LEAKAGE: Unfiltered sys_* shapes expose other users' definitions** | **CRITICAL**: Backend MUST filter ElectricSQL shapes by user role; RLS policies enforce row-level security; audit logs track unauthorized access attempts |
| ElectricSQL is production-ready but relatively new ecosystem | Start with Phase 1 (read-only sys_* access); validate in staging before enabling mutations |
| PostgreSQL logical replication must be configured | Template includes migration to enable replication; documented in SETUP.md |
| Network disconnection during mutation | ElectricSQL queues mutations; replay on reconnect; UI shows "syncing..." indicator |
| Conflict resolution (rare) | UI shows conflict alert; default is last-write-wins; can extend with manual resolution UI in Phase 2 |
| Local SQLite grows large | Implement `VACUUM` on app startup; set `maxLocalStorageMB` in ElectricSQL config |
| Type mismatch between server schema and local sync | Use generated Zod schemas from backend types; validate on both sides |
| Double-write prevention | ElectricSQL enforces single-writer principle; document in code comments |
| **Offline mutation of unauthorized data** | Client-side optimistic writes still respect RLS; server rejects mutations if user loses access before sync completes |

## Pre-Deployment Security Checklist

**BEFORE deploying any generated app with ElectricSQL to production, verify these security requirements:**

### Database Schema
- ✅ `sys_table` has columns: `created_by`, `role_accessible[]`, `is_public`
- ✅ `sys_column` has columns: `created_by`, `role_accessible[]`, `is_public`
- ✅ `sys_field` has columns: `created_by`, `role_accessible[]`, `is_public`
- ✅ `sys_rule` has columns: `created_by`, `role_accessible[]`, `is_public`
- ✅ All sys_* tables have Row Level Security (RLS) enabled
- ✅ RLS policies enforce filtering by `created_by` and `role_accessible`

### Backend API Security
- ✅ `/api/electric-sql/shapes` endpoint validates user identity before returning shape queries
- ✅ Endpoint filters shapes by user's actual roles (not user-provided roles)
- ✅ All shape queries include WHERE clauses restricting by `created_by`, `role_accessible`, or `is_public`
- ✅ **NEVER return unfiltered shape queries** (`SELECT * FROM sys_field` is FORBIDDEN)
- ✅ Shape endpoint returns 403 if userId doesn't match authenticated user

### Frontend Initialization
- ✅ `setupElectricSync()` calls `/api/electric-sql/shapes` endpoint (not local shape building)
- ✅ ElectricSQL client receives filtered shape queries, not raw table names
- ✅ Frontend never hardcodes ElectricSQL shape subscriptions
- ✅ Authentication token is required and validated before sync begins

### Testing & Validation
- ✅ **Multi-role test**: Log in as Role A, verify cannot see Role B's custom fields
- ✅ **Role removal test**: Remove user from role while offline, verify mutations rejected on sync
- ✅ **Unauthed access test**: Attempt direct ElectricSQL connection without auth token (should fail)
- ✅ **Privilege escalation test**: Try to access other user's private definitions (should fail)
- ✅ **Audit trail test**: Verify unauthorized access attempts are logged

### Deployment Steps
1. Enable RLS on all sys_* tables in PostgreSQL
2. Create RLS policies for each sys_* table
3. Deploy `/api/electric-sql/shapes` endpoint with validation logic
4. Configure ElectricSQL server with authentication middleware
5. Run security checklist tests in staging
6. Enable audit logging for sys_* table access
7. Document shape filtering in generated app's SECURITY.md
8. Disable direct ElectricSQL client connections (only allow via authenticated backend)

### Audit & Monitoring
- ✅ Enable PostgreSQL audit logging for sys_* tables
- ✅ Monitor unauthorized shape requests in backend logs
- ✅ Alert on repeated failed authorization attempts
- ✅ Track who accessed which user's definitions in audit_log table
- ✅ Monthly security review of shape filtering logic
